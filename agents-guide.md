# Agents 说明文档

> 本文档整理项目所有 Agent 的提示词要点、触发方式与工作逻辑。

---

## 一、Agent 总览

| Agent | 类型 | 模型 | 权限 | 核心职责 |
|-------|------|------|------|---------|
| **plan** | subagent | glm-5.1 | 读写+执行 | 调研代码库，输出开发计划 |
| **annotate** | subagent | glm-5.1 | 编辑+禁止 bash | 批注审查计划，推动完善 |
| **coder** | subagent | glm-5.1 | 读写+执行 | 按 plan 逐项实施代码 |
| **review** | subagent | glm-5.1 | 读写+执行 | 逐项验收，创建 review 文档 |
| **explore** | 内置 | — | 只读 | 快速搜索代码库 |
| **general** | 内置 | — | 读写+执行 | 通用多步骤任务 |

### 配置层级

| 位置 | 路径 | 说明 |
|------|------|------|
| **全局** | `~/.config/opencode/agents/*.md` | 所有项目可用，个人通用 |
| **项目** | `.opencode/agents/*.md` | 仅当前项目，团队共享，同名覆盖全局 |

Agent 和 Skill 在全局与项目中各放一份，项目级跟随 git 团队共享，全局级个人独立项目自动可用。

## 二、Plan Agent — 计划制定

### 触发方式

- 用户要求制定计划、规划功能、编写 plan、技术方案设计时触发
- 对应 `dev-workflow.md` Step 1-2

### 不要触发

- 代码实施、批注审查、review 收尾、纯调研

### 工作逻辑

```
用户需求
   │
   ▼
Step 1 · 调研
   │  搜索代码库结构、理解架构、查阅 MEMORY.md
   │  产出：plans/research/research-{功能名/版本号}.md
   │
   ▼
Step 2 · 规划
   │  选择模板（Swift Feature / Swift Refactor）
   │  填充模板各章节
   │  产出：plans/plan-{功能名}-v{版本号}.md
   │
   ▼
完成 → 等待用户审核（不执行代码）
```

### Plan 必须包含的章节

1. **Skill 调用要求**（零章节）— 实施时需加载的 skill 列表
2. **需求背景** — 背景、功能需求、非功能需求
3. **技术方案** — 架构、目录结构、数据模型、核心流程、设计决策
4. **UI 设计**（如适用）
5. **实现步骤** — 每步含文件路径、动作、验证方式
6. **不修改的内容** — 明确边界
7. **验证要点** — 功能验证 + 回归验证
8. **文件变更清单** — 新建/修改文件及行数估算
9. **阶段跟踪** — 每步状态，初始全为 ⬜

### 提示词示例

> **原则**：说清「做什么」和「边界」，不说「怎么做」（实现方式留给 plan）。
> 一句话太模糊，Agent 会猜需求，容易方向跑偏浪费批注轮次。

**新功能规划（推荐格式）：**

```
帮我规划「一键排除专业」功能：
- 用户在专业列表页可勾选多个专业
- 点击排除后从候选列表移除
- 排除的专业可恢复
- 需要持久化存储排除状态
```

**其他场景：**

| 场景 | 提示词 |
|------|--------|
| 重构规划 | `规划 swift-log 集成重构：替换所有 print/os_log 为 swift-log，保持现有日志点不遗漏，新增结构化日志` |
| 带版本号 | `给 v1.45.1 版本规划：志愿筛选增加按院校层次过滤，支持多选，结果实时刷新` |
| 只调研 | `先调研项目中日志模块的现状，输出 research 文档` |
| 指定模板 | `用 Swift Feature 模板规划这个功能` |
| 追加需求 | `在现有 plan 基础上，补充离线缓存的需求：无网络时展示缓存数据，恢复网络后自动同步` |

### 关键约束

- Plan 状态设为 `规划中`
- 完成后**不执行代码**，等待审核
- 不自动加载 Skill（Skill 仅实施阶段加载）
- 使用 Xcode MCP 工具读取项目文件

## 三、Annotate Agent — 批注循环

### 触发方式

- 用户要求批注 plan、审查计划、纠正假设、补充约束、画流程图解释、标记 don't implement yet 时触发
- 对应 `dev-workflow.md` Step 3

### 不要触发

- 代码实施、review 收尾、调研、plan 编写

### 批注类型

| 类型 | 标记格式 | 说明 |
|------|---------|------|
| 纠正错误假设 | `> 🔴 批注：{内容}` | 指出不符合实际的地方 |
| 否决不合理方案 | `> 🔴 否决：{原因}` | 明确某方案不可行 |
| 补充约束条件 | `> 🟡 补充：{约束}` | 添加技术或业务约束 |
| 画流程图解释 | `> 🔵 流程图：{图}` | 复杂逻辑用 ASCII 流程图解释 |
| 标记暂不实施 | `> ⏸️ don't implement yet：{原因}` | 标记不实施的部分 |

### 工作逻辑

```
读取 plan.md
   │
   ▼
逐项审查（四个维度）
   │  需求维度：完整性、优先级、非功能需求
   │  技术维度：架构、数据模型、核心流程、并发安全
   │  实施维度：路径准确性、步骤顺序、Skill 声明
   │  风险维度：破坏性、边缘情况、兼容性
   │
   ▼
写入批注
   │  在 plan.md 对应位置插入批注标记
   │
   ▼
用户决策
   │  ✅ 采纳 / ❌ 不同意 / ✏️ 调整 / ⏭️ 跳过
   │  （内联决策 或 对话回复）
   │
   ▼
处理决策 → 原地清理 plan.md
    │
    ├── 还有疑虑 → 继续批注（1-6 轮循环）
    │
    └── 没有问题 → 状态改为「实施中」，可以放手执行
         - 采纳的修改已融入正文
         - 批注标记和汇总章节已删除
         - don't implement yet 内容已删除
```

### 循环轮次

| 轮次 | 重点 |
|------|------|
| 第 1 轮 | 发现明显问题 |
| 第 2 轮 | 深入细节，补充约束 |
| 第 3-4 轮 | 边缘情况、异常处理 |
| 第 5-6 轮 | 最终确认，确保无遗漏 |

### 决策标记

| 标记 | 含义 | Agent 处理 |
|------|------|-----------|
| ✅ 采纳 | 同意批注，按批注改 | 内容融入正文，移除标记 |
| ❌ 不同意 | 否定批注，保持原方案 | 移除标记，保留原方案 |
| ✏️ 调整 | 部分接受 | 按调整修改正文，移除标记 |
| ⏭️ 跳过 | 知道了但本轮不处理 | 批注保留，下轮重新评估 |

### 提示词示例

**触发批注：**

> **原则**：指定 plan 路径 + 审查方向，比笼统的「审查一下」更高效。
> 不指定方向时 Agent 会全量审查，指定方向可聚焦、减少无关批注。

**可用审查方向：**

| 方向 | 说明 | 适用场景 |
|------|------|---------|
| 需求完整性 | 遗漏场景、边界条件、优先级 | 需求刚理清，怕有遗漏 |
| 架构合理性 | 模块划分、依赖关系、扩展性 | 新模块或架构变更 |
| 数据模型 | 字段完整性、关系设计、持久化方案 | 涉及数据存储 |
| 并发安全 | 线程安全、数据竞争、actor 隔离 | 多线程/异步操作 |
| 性能 | 时间复杂度、内存占用、批量操作 | 大数据量、高频操作 |
| API 兼容性 | 接口变更、向后兼容、迁移策略 | 改动已有公开接口 |
| 实施可行性 | 路径准确性、步骤顺序、依赖关系 | plan 即将执行 |
| 边缘异常 | 空值、网络失败、权限拒绝 | 需要健壮性保障 |

**提示词示例：**

| 场景 | 提示词 |
|------|--------|
| 启动批注 | `批注 plans/plan-onekey-major-exclusion-v1.md` |
| 指定审查方向 | `批注这个 plan，重点看数据模型和并发安全` |
| 多方向组合 | `批注这个 plan，看架构合理性 + API 兼容性` |
| 画流程图 | `plan 里专业排除的流程有点绕，画个流程图解释一下` |
| 补充约束 | `批注时注意补充离线场景的约束条件` |
| 标记暂不实施 | `批注这个 plan，第三方登录部分标记 don't implement yet` |
| 多轮深入 | `上一轮批注已处理，再审一遍，重点看边缘异常` |

**决策回复（对话方式）：**

| 场景 | 提示词 |
|------|--------|
| 全部采纳 | `没问题，下一轮` |
| 部分采纳 | `🔴 第3条我不同意，原因是… 🟡 第5条调整成… 其他 OK` |
| 推翻否决 | `否决那条不用换方案，按原方案来，因为…` |
| 补充新发现 | `再加一条：这里要考虑离线场景` |
| 直接拍板执行 | `执行` — 状态改实施中，启动实施 |

**决策回复（内联方式，直接写在 plan 中）：**

```
> 🔴 批注：UserDefaults 不支持线程安全，多线程写入会丢数据
> ✅ 采纳 — 改用 Keychain + actor

> 🔴 否决：用遍历查找性能太差
> ❌ 不同意 — 数据量小，遍历够用

> 🟡 补充：需要考虑离线场景
> ✏️ 调整 — 本期只做缓存读取，离线写入放 v2

> ⏸️ don't implement yet：OAuth 第三方登录
> ✅ 同意
```

### 关键约束（Annotate）

- **只做批注，不做代码实施**
- Plan 状态保持 `规划中`
- 批注指向具体位置，不笼统评论
- 使用 Xcode MCP 工具验证代码库现状

## 三-A、Coder Agent — 计划实施

### 触发方式

- 用户要求执行 plan、实施代码、implement it all、mark completed 时触发
- 对应 `dev-workflow.md` Step 4

### 不要触发

- plan 编写、批注审查、review 收尾、调研

### 工作逻辑

```
读取 plan.md
    │
    ▼
前置检查
    │  确认 Skill 列表
    │  读取 AGENTS.md 了解项目约束
    │
    ▼
加载 Skill（强制）
    │  按 plan 零章节声明逐个加载
    │  coding-standards / naming-conventions / urouter / swift-log 等
    │
    ▼
逐项实施（按 Step 顺序）
    │  ┌─ 读取 Step 内容 ──────────────────────┐
    │  │  使用 Xcode MCP 工具读取/修改/创建文件  │
    │  │  实现 Step 中的所有子任务               │
    │  │  编译验证 → 不通过则修复再继续          │
    │  │  SwiftLint 检查 + 自动修复              │
    │  │  标记 ✅ 已完成                         │
    │  └───────────────────────────────────────┘
    │         ↓ 重复直到所有 Step 完成
    │
    ▼
输出实施报告
    │  Step 完成度 + 偏差记录 + 三板斧结果
    │
    ▼
完成 → 等待用户指令（不自动启动 Review）
```

### 实施原则

| 原则 | 说明 |
|------|------|
| implement it all | 把计划里的所有内容全部实现 |
| mark completed | 每完成一项就标记为已完成 |
| don't stop | 中间不要停，一口气做完 |
| 每步编译 | 完成 Step 后必须编译验证，不过则修复再继续 |
| 先读后写 | 修改文件前必须先读取原内容 |
| 保留原有注释 | 他人编写的注释严禁删除或篡改 |
| 新增代码必须注释 | 新增类/属性/方法均须添加 `///` 文档注释 |

### 提示词示例

| 场景 | 提示词 |
|------|--------|
| 启动实施 | `执行 plans/plan-swift-log-integration-v1.md` |
| 继续实施 | `implement it all, mark completed, don't stop` |
| 从某步开始 | `从 Step 3 开始执行这个 plan` |

### 关键约束（Coder）

- **只做实施，不做 plan 编写、批注、review**
- 严格按 plan 步骤顺序执行，不跳步
- 每步必须编译验证通过才标记完成
- 不自动 git 提交
- 不自动启动 Review
- 使用 Xcode MCP 工具操作代码

## 四、Review Agent — 审核与 Review 收尾

### 触发方式

- 用户要求审核 plan、review 收尾、验收检查、遗留问题管理、签收判定时触发
- 对应 `dev-workflow.md` Step 6

### 不要触发

- 代码实施、plan 编写、调研

### 工作逻辑

```
读取 plan.md + 检查代码库实施结果
   │
   ▼
运行三板斧
   │  1. 编译：xcodebuild ... quiet build → 无错误
   │  2. Lint：swiftlint lint --path YouZhiYuan → 无新增 warning
   │  3. 测试：按实际方案执行 → 通过
   │
   ▼
创建 Review 文档
   │  产出：reviews/review-{功能名}-v{版本号}.md
   │  必含章节：验收总览、Plan 偏差记录、遗留问题、经验总结、签收
   │
   ▼
问题分级与双向同步
   │  遗留问题 P0/P1/P2 分级
   │  review 是 source of truth → 同步回 plan 阶段跟踪表
   │
   ▼
签收判定
   │
   ├── P0/P1 未关闭 → 打回（Plan 状态：review 打回）
   │                  修复后重新 review
   │
   └── P0/P1 已关闭 → 签收（Plan 状态：已完成）
                      经验沉淀 → MEMORY.md
```

### 问题分级

| 级别 | 含义 | 签收条件 |
|------|------|---------|
| 🔴 P0 | 功能不可用、数据错误、严重 bug | 必须修完才能签收 |
| 🟠 P1 | 功能不完整、验收项未通过 | 必须关闭或转入新 plan |
| 🟡 P2 | 优化项、边缘场景、技术债 | 可遗留，注明后续打算 |

### 签收条件

- [ ] 所有 P0 问题已关闭
- [ ] 所有 P1 问题已关闭或已转入新 plan 跟踪
- [ ] 三板斧（编译 / Lint / 测试）均通过

### 双向同步机制

```
reviewer 发现问题
    ├─ 1. 在 review「遗留问题」表记录（主记录）
    ├─ 2. 同步回 plan：阶段状态 → ⚠️ review 打回，备注问题编号
    └─ 3. 修复后双向更新：
           review：⬜ → ✅ 已修复
           plan：⚠️ → ✅ 已完成（review 后修复）
```

### 提示词示例

> **原则**：指定 plan 路径，可选加关注重点，让 Agent 验收更有针对性。

| 场景 | 提示词 |
|------|--------|
| 启动 review | `review plans/plan-onekey-major-exclusion-v1.md` |
| 指定重点 | `review 这个 plan，重点关注数据持久化那步的实现` |
| 只跑三板斧 | `跑一下编译、lint、测试` |
| 查看遗留问题 | `review-v1 里有哪些遗留问题还没关？` |
| 签收判定 | `这个 plan 可以签收了吗？` |
| 代码审查 | `对这次改动做代码审查，重点看命名规范和 MARK 分段` |

### 关键约束（Review）

- **只做发现和记录，不做实施**
- review 是问题的 source of truth
- Plan 状态设为 `review 中`；打回时设为 `review 打回`
- 使用 Xcode MCP 工具检查代码

## 五、Explore Agent — 代码库探索

### 触发方式

- 需要快速搜索文件、查找代码关键词、回答代码库结构问题时触发
- 支持三种彻底度：quick / medium / very thorough

### 工作逻辑

1. 按指定的彻底度搜索代码库
2. 通过 glob 模式查找文件
3. 通过 regex 搜索代码内容
4. 汇总发现并回答问题

### 提示词示例

| 场景 | 提示词 |
|------|--------|
| 查找类定义 | `URouter 类在哪里定义的？` |
| 搜索引用 | `哪些文件引用了 URouter？` |
| 了解架构 | `项目的路由是怎么组织的？` |
| 查找模式 | `找出所有遵循 Routable 协议的类` |
| 定位 bug | `搜索所有 force unwrap 的地方` |
| 快速确认 | `News 模块下有哪些文件？` |

彻底度可通过措辞控制：
- 快速：`快速找一下 XXX 在哪`
- 中等：`帮我看看 XXX 相关的代码`
- 深入：`彻底搜索整个项目中所有跟 XXX 相关的内容`

### 适用场景

- "这个类在哪里定义的？"
- "哪些文件引用了 XXX？"
- "项目中的路由是怎么组织的？"

## 六、General Agent — 通用任务

### 触发方式

- 需要执行多个并行工作单元、研究复杂问题、执行多步骤任务时触发

### 工作逻辑

1. 接收任务描述
2. 可并行执行多个子任务
3. 汇总结果返回

### 提示词示例

| 场景 | 提示词 |
|------|--------|
| 并行调研 | `同时调研 News 模块和 Profile 模块的数据层实现` |
| 复杂任务 | `把这三个模块的日志都迁移到 swift-log，输出对比报告` |
| 多步骤操作 | `清理 plans/ 下已完成的 plan，归档到 reviews/，输出摘要` |
| 批量处理 | `给这三个文件都加上 MARK 分段注释` |

### 适用场景

- 需要同时调研多个模块
- 复杂的多步骤操作
- 需要并行处理的任务

## 七、协作关系

```
用户需求
   │
   ▼
┌─ Plan Agent ─────────┐
│  调研 → 输出 plan.md  │
└────────┬─────────────┘
         │
         ▼
┌─ Annotate Agent ──────────────────┐
│  批注循环（1-6 轮）                │
│  用户决策 → 原地清理 plan.md       │
│  状态改为实施中                    │
└────────┬──────────────────────────┘
          │
          ▼
┌─ Coder Agent ──────────────────────┐
│  加载 Skill → 按 plan 实施          │
│  implement it all / mark completed │
└────────┬──────────────────────────┘
         │
         ▼
┌─ Review Agent ────────────────────┐
│  三板斧 → 创建 review 文档         │
│  问题分级 → 双向同步               │
│  签收 / 打回                      │
└───────────────────────────────────┘
```

**辅助 Agent：**

- **Explore** — 任何阶段需要搜索代码时调用

## 八、状态生命周期

```
规划中 → 实施中 → review 中 → 已完成
                ↑          │
                │  review 打回
                └──────────┘
```

| 状态 | 触发者 | 含义 |
|------|-------|------|
| 规划中 | Plan Agent | plan 还在写，未开始实施 |
| 实施中 | Coder Agent 开始执行 plan | 正在按阶段开发 |
| review 中 | Review Agent | 实施完成，等待或正在 review |
| review 打回 | Review Agent（发现 P0/P1） | 需回去修复 |
| 已完成 | Review Agent（签收通过） | 所有 P0/P1 问题已关闭 |

## 九、端到端流程提示词示例

以「一键排除专业」功能为例，展示从调研到签收的完整提示词：

```
# Step 1 · 调研
> 调研一下专业筛选模块的代码现状，输出 research 文档

# Step 2 · 规划
> 帮我规划「一键排除专业」功能：
> - 用户在专业列表页可勾选多个专业
> - 点击排除后从候选列表移除
> - 排除的专业可恢复
> - 需要持久化存储排除状态

# Step 3 · 批注（第 1 轮）
> 批注 plans/plan-onekey-major-exclusion-v1.md

# Step 3 · 批注（决策）
> 🔴 第3条我不同意，其他 OK，下一轮

# Step 3 · 批注（第 2 轮）
> 再审一遍，重点看边缘情况

# Step 3 · 批注（拍板执行）
> 执行

# Step 4 · 放手执行
> implement it all, mark completed, don't stop

# Step 5 · 测试
> 安排测试并出测试报告

# Step 6 · Review 收尾
> review plans/plan-onekey-major-exclusion-v1.md
```
